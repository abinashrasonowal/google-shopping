# Google Shopping Scrapers

Three independent Apify actors for scraping Google Shopping, written in JavaScript (Node.js, ESM). Each lives in its own directory and is deployed separately.

**Run these actors directly on Apify:**
- 🛒 [Google Shopping Search Scraper](https://apify.com/breathtaking_turtle/google-shopping)
- 🔎 [Google Shopping Product Details Scraper](https://apify.com/breathtaking_turtle/google-shopping-immersive)

| Directory | Actor | Input | Output |
|---|---|---|---|
| [`shopping/`](shopping/) | Search-results scraper | `query` + `country` | List of ~40 product listings |
| [`immersive/`](immersive/) | Product-detail scraper | product `url` (with `prds=`) + `country` | Single product: specs, sellers, filters |
| [`product-resolution/`](product-resolution/) | Resolution engine | retailer `url` + `country` | Matching Google Shopping offer for that product |

This is a **pnpm workspace** of cross-depending modules (like a Gradle multi-module
build). Each module is a package; modules call each other's exported functions
directly, and esbuild **bundles** each actor's dependency graph into a single
self-contained `dist/main.js` so it stays independently deployable.

```
client/                  @gs/client      base module: ProxyHttpClient + isBlocked
shopping/    (actor)      @gs/shopping    depends on @gs/client; EXPORTS runSearch/parseProducts/…
immersive/   (actor)      @gs/immersive   depends on @gs/client; EXPORTS parseProduct
product-resolution/ (actor)              depends on @gs/client + @gs/shopping + @gs/immersive
```

Dependency graph (a DAG — no cycles):

```
@gs/client  ←  @gs/shopping  ←┐
@gs/client  ←  @gs/immersive ←┤
@gs/client  ←────────────────  product-resolution
```

Each actor directory:

```
<actor>/
├── .actor/            # actor.json, input/output/dataset schemas, openapi
├── src/
│   ├── main.js        # Actor lifecycle / orchestration (the deployable entry)
│   ├── search.js | immersive.js   # the module's EXPORTED library (shopping / immersive)
│   └── ...            # actor-specific modules (e.g. product-resolution: meta/query/match)
├── dist/              # GENERATED by esbuild (bundle + runtime-only package.json); git-ignored
├── Dockerfile         # installs dist/package.json deps, runs dist/main.js
└── package.json       # name @gs/<actor>, runtime deps, workspace devDeps it consumes
```

## Modules & direct reuse

- **`@gs/client`** ([client/](client/)) — base module used by everyone: `ProxyHttpClient`
  (Apify-proxy fetch with retries) and `isBlocked`.
- **`@gs/shopping`** — owns the search logic and **exports** `runSearch` / `searchProducts`
  / `parseProducts` / `buildSearchUrl` ([shopping/src/search.js](shopping/src/search.js)).
- **`@gs/immersive`** — owns and **exports** `parseProduct` ([immersive/src/immersive.js](immersive/src/immersive.js)).
- **product-resolution** imports those directly, e.g.:
  ```js
  import { ProxyHttpClient, isBlocked } from '@gs/client';
  import { runSearch }   from '@gs/shopping';
  import { parseProduct } from '@gs/immersive';
  ```

There is **no separate "shared" grab-bag** — each piece of logic lives in the module
that owns it, and consumers depend on that module (the Gradle `implementation
project(':client')` pattern). Importing `@gs/shopping` resolves to its library
(`src/search.js`), never its actor `main.js`, so no `Actor.init()` runs on import.

```bash
pnpm install     # once, from the repo root — links @gs/shared into each actor
pnpm build       # bundle all actors  (or: pnpm build:shopping / :immersive / :resolution)
```

`pnpm build` runs [`scripts/build.mjs`](scripts/build.mjs): esbuild inlines `@gs/shared`
into `<actor>/dist/main.js` (keeping `apify`/`cheerio`/`got-scraping` external) and writes
a clean `<actor>/dist/package.json` with only the runtime deps — no `workspace:` specs,
which npm (used in the Docker image) rejects.

## Running & deploying

```bash
pnpm install                 # once

# local dev — runs src/ directly via the workspace symlink (no build needed)
cd shopping && pnpm dev      # or: apify run

# deploy — builds the bundle, then apify push (from the repo root)
pnpm deploy:shopping
pnpm deploy:immersive
pnpm deploy:resolution
```

`dist/` is git-ignored but force-included for upload via each actor's `.actorignore`
(`!dist/`), so `apify push` ships the bundle while the source tree stays clean.

## Proxy requirements

- **shopping** uses `GOOGLE_SERP` proxies.
- **immersive** uses `RESIDENTIAL` proxies.
- **product-resolution** uses both `RESIDENTIAL` (meta + immersive) and `GOOGLE_SERP` (search).

Make sure the relevant proxy access is enabled on your Apify account.
